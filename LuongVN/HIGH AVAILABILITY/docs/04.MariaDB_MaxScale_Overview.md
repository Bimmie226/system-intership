# MariaDB MaxScale 

MariaDB MaxScale là một database proxy có chức năng chuyển tiếp các câu lệnh cơ sở dữ liệu đến một hoặc nhiều database server 

Việc chuyển tiếp này được thực hiện dựa trên các rule được xây dựng dựa trên khả năng hiểu ngữ nghĩa của câu lệnh SQL, và dựa trên vài trò của các server trong cluster 

MariaDB MaxScale được thiết kế để cung cấp các chức năng load balancing và high availability đối với ứng dụng 

MariaDB MaxScale có kiến trúc linh hoạt và có khả năng mở rộng, với các thành phần plugin để hỗ trợ nhiều giao thức và nhiều phương pháp routing khác nhau.

Nhiều dịch vụ của MariaDB MaxScale được triển khai dưới dạng module, ta có thể tạo thêm các module bổ sung để hoạt động với MariaDB MaxScale. Các loại Module thường dùng: 
- `Protocol Module`: triển khai giao tiếp giữa client <-> MariaDB MaxScale <-> DB Server 
- `Routing Module`: phân tích query từ client và quyết định db nào sẽ nhận query đó. Quyết định thường dựa trên routing rules và trạng thái của db server.
- `Filter Module`: xử lý dữ liệu khi nó đi qua MariaDB MaxScale. Filter thường dùng để log query hoặc chỉnh sửa phản hồi từ server.

## I. Overview 

MariaDB MaxScale là một proxy, router và load balancer nâng cao: 
- MaxScale thực hiện failover tự động cho MariaDB replication. Khi primary server bị lỗi, MaxScale sẽ promote 1 replica thành primary mới và chuyển hướng các replica còn lại về primary mới này 
- Router `ReadWriteSplit` của MaxScale thực hiện cân bằng tải dựa trên query. `ReadWriteSplit` chuyển tất cả câu lệnh ghi (write statement) đến primary server hiện tại và phân phối các câu lệnh đọc (read statement) đến các replica server.
- Router `ReadConnRoute` thực hiện cân bằng tải dựa trên connection. `ReadConnRoute` chuyển mỗi kết nối đến một node primary hoặc replica duy nhất, tùy theo cấu hình 
- MaxScale cung cấp các cơ chế tích hợp sẵn để thực hiện bảo trì server mà không làm gián đoạn ứng dụng hoặc client. Server có thể được đưa vào chế độ maintainance bằng command-line interface với `MaxCtrl`, giao diện web MaxGUI hoặc REST API 
- Filter Cache của MaxScale có thể cải thiện hiệu năng SELECT bằng cách cache và tái sử dụng kết quả.

## II. MaxScale Configuration 

### 2.0 Các khái niệm 

- `Connection routing`: định tuyến theo kết nối. MaxScale chập nhận kết nối từ Client và chuyển dữ liệu trên kết nối đó đến 1 db duy nhất thông qua 1 kết nối duy nhất 
- `statement routing`: định tuyến theo câu lệnh. 
- `module`: Module là một thành phần code riêng biệt có thể được load động vào MariaDB MaxScale để tăng thêm chức năng.
- `connection failover`: Khi một kết nối giữa MariaDB MaxScale và database server bị lỗi, một kết nối thay thế sẽ tự động được tạo sang server khác bởi MariaDB MaxScale mà không cần client can thiệp.
- `backend database`: Thuật ngữ dùng để chỉ database nằm phía sau MariaDB MaxScale và được ứng dụng truy cập thông qua MariaDB MaxScale.
- `REST API`: giao diện quản trị thông qua HTTP 

### 2.1 Objects 

**Server**

Server đại diện cho một database server riêng lẻ mà client có thể kết nối thông qua MariaDB MaxScale.

Trạng thái của server có thể thay đổi trong suốt vòng đời của nó và thường được cập nhật bởi một monitor. Tuy nhiên, cũng có thể cập nhật thủ công.

**Monitor**

Monitor module có khả năng giám sát trạng thái của 1 loại cluster cụ thể và cung cấp trạng thái đó cho router của MaxScale 

- `mariadbmon`: giám sát primary-replica cluster 
- `galeramon`: giám sát Galera cluster 

Monitor module có section riêng trong file cấu hình MaxScale.

**Filter**

Filter module nằm phía trước router trong chuỗi xử lý request của MaxScale.

Filter có section riêng trong file cấu hình và được tham chiếu từ service.

**Router**

Router module chịu trách nhiệm định tuyến request đến backend server dựa trên đặc điểm của request và/hoặc thuật toán routing.

**Service**

Service trừu tượng hóa 1 database cluster, client sẽ thấy cụm này chỉ như 1 service db duy nhất 

Service có section riêng trong file cấu hình MaxScale.

**Listener**

Listener định nghĩa port mà MaxScale sẽ lắng nghe.

## IV. Configuration 

File cấu hình mặc định của MaxScale: `/etc/maxscale.cnf`

Ví dụ 1 file cấu hình: 

```ini
[MyService]
type=service
router=readconnroute
servers=server1,
        server2,
        server3
```

### 4.0 Cách đặt tên cho Section 

Tên section không được chứa khoảng trắng và không được bắt đầu bằng ký tự `@@`.

Chỉ được phép dùng: 
- chữ cái và số (a-z, A-Z, 0-9) 
- các ký tự đặc biệt: `_`, `.`, `~`, `-`

### 4.1 Cấu hình động 

Các thay đổi runtime sẽ Overwrite cấu hình tương ứng của Object đó 

Để tắt việc lưu thay đổi runtime và làm cho các thay đôi runtime trở nên không bền vững: 
- `persist_runtime_changes=false`
- `load_persisted_configs=false`

Trong section `[maxscale]`

Sau khi config như trên, các thay đổi runtime sẽ không được lưu xuống disk và sẽ mất khi restart process

### 4.2 Các kiểu tham số đặc biệt 

**Boolean**

- true: `true`, `yes`, `on`, `1`
- false: `false`, `no`, `off`, `0`

**Sizes**

- IEC: Ki, Mi, Gi, Ti (theo 1024) 
- SI: k, M, G, T (theo 1000) 

Không phân biệt hoa thường 

**Duration(thời lượng)**

- h: giờ 
- m/min: phút
- s: giây
- ms: mili giây 

**Percent**

```ini
some_param=42%
```

**Regex**

```ini
match=/select/
match=/^select/
match=/select$/
```


**Enum**

Enum có tập giá trị cố định: 
- `enum`: chỉ 1 giá trị
- `enum_mask`: nhiều giá trị, ngăn cách bằng dấu phẩy

```ini
router_options=master,slave
```

**Path list**

Kiểu pathlist nhận một hoặc nhiều đường dẫn filesystem, phân tách bằng dấu `:`, không được có khoảng trắng 

```ini
path_list_parameter=/tmp/something.log:/var/log/maxscale/maxscale.log
```

### 4.3 Global Settings

```ini
# =========================
# GLOBAL SETTINGS
# =========================
[maxscale]

threads=auto
logdir=/var/log/maxscale
datadir=/var/lib/maxscale
piddir=/run/maxscale

log_warning=true
log_notice=true
log_info=false
log_debug=false

log_throttling=10,1s,10s
skip_name_resolve=true

# Giảm DNS lookup → tăng performance
host_cache_size=256

# Tăng an toàn auth backend
auth_connect_timeout=10s

# =========================
# REST API (ADMIN)
# =========================
admin_host=0.0.0.0
admin_port=8989
admin_auth=true

# Bật HTTPS (production bắt buộc nên dùng)
admin_ssl_key=/etc/maxscale/certs/server.key
admin_ssl_cert=/etc/maxscale/certs/server.crt
admin_secure_gui=true

admin_log_auth_failures=true

# Chỉ cho phép nội bộ hoặc bastion host
admin_readwrite_hosts=10.0.0.10,127.0.0.1

# =========================
# MONITOR (MySQL/MariaDB cluster)
# =========================
[Galera-Monitor]
type=monitor
module=mariadbmon
servers=mariadb1,mariadb2,mariadb3
user=maxscale_mon
password=mon_password
monitor_interval=2s
auto_failover=true
auto_rejoin=true

# =========================
# SERVERS
# =========================
[mariadb1]
type=server
address=10.0.0.11
port=3306
protocol=MariaDBBackend

[mariadb2]
type=server
address=10.0.0.12
port=3306
protocol=MariaDBBackend

[mariadb3]
type=server
address=10.0.0.13
port=3306
protocol=MariaDBBackend

# =========================
# SERVICES
# =========================

# Read/Write splitting service
[RW-Service]
type=service
router=readwritesplit
servers=mariadb1,mariadb2,mariadb3
user=maxscale_rw
password=rw_password

# Read-only service (replicas)
[RO-Service]
type=service
router=readconnroute
router_options=slave
servers=mariadb2,mariadb3
user=maxscale_ro
password=ro_password

# =========================
# LISTENERS
# =========================

# Client endpoint (read/write)
[RW-Listener]
type=listener
service=RW-Service
protocol=MariaDBClient
port=4006

# Read-only endpoint
[RO-Listener]
type=listener
service=RO-Service
protocol=MariaDBClient
port=4008
```

### 4.4 Service 

1 service đại diện cho dịch vụ db mà MariaDB MaxScale cung cấp cho client. 

1 service gồm một tập hợp các db backend server và 1 thuật toán routing quyết định cách MaxScale gửi câu lệnh hoặc kết nối tới các backend đó 

Một service có thể được xem như một `database ảo` mà MaxScale cung cấp cho client.

```ini
[Test-service]
type=service
```

### 4.5 router 
Tham số router xác định tên module router dùng để thực hiện thuật toán routing giữa client và backend database.

Ngoài ra, router có thể nhận thêm danh sách option (phân tách bằng dấu phẩy) để điều khiển hành vi routing.

```ini
router=readconnroute
router_option=master,slave
```

### 4.6 targets
Danh sách server hoặc service mà service này sẽ route tới 

```init
targets=my-service,server2
```

### 4.7 servers
Dánh sách backend server:

```ini
servers=server1,server2,server3
```

### 4.8 cluster
Chỉ định monitor quản lý các server:

```ini
cluster=galeramon
```

### 4.9 include
include section dùng để tái sử dụng config chung: 

Thay vì viết từng config service như này: 

```ini
[Monitor1]
type=monitor
module=mariadbmon
user=the_user
password=the_password
handle_events=false
monitor_interval=2000ms
backend_connect_timeout = 3s
backend_connect_attempts = 5
servers=Server1, Server2

[Monitor2]
type=monitor
module=mariadbmon
user=the_user
password=the_password
handle_events=false
monitor_interval=2000ms
backend_connect_timeout = 3s
backend_connect_attempts = 5
servers=Server3, Server4
```

Ta có thể tạo 1 config service chung và áp dụng cho các service khác như sau:

```ini
[Monitor-Common]
type=include
module=mariadbmon
user=the_user
password=the_password
handle_events=false
monitor_interval=2000ms
backend_connect_timeout = 3s
backend_connect_attempts = 5

[Monitor1]
type=monitor
@include=Monitor-Common
servers=Server1, Server2

[Monitor2]
type=monitor
@include=Monitor-Common
servers=Server3, Server3
```
