# Upgrade Kubernetes Version

Kubernetes upgrade là quá trình cập nhật hệ thống Kubernetes lên phiên bản mới hơn. Đây là một tác vụ phức tạp, đòi hỏi phải được lập kế hoạch và thực hiện một cách cẩn thận. Quá trình nâng cấp có thể bao gồm việc cập nhật các thành phần như Kubernetes API Server, Controller Manager, Scheduler, Kubelet, Kube-Proxy, ... 

Điều quan trọng là việc nâng cấp Kubernetes không chỉ đơn thuần là chuyển sang một phiên bản mới hơn. Mục tiêu của quá trình này là đảm bảo toàn bộ hệ thống vẫn tiếp tục hoạt động ổn định sau khi nâng cấp, không gây gián đoạn với các ứng dụng đang chạy. Do đó, cần cân bằng giữa việc tận dụng các tính năng mới, các bản vá bảo mật với yêu cầu duy trì tính ổn định và độ tin cậy của hệ thống 

# Upgrading kubeadm Cluster 

Nâng cấp 1 cụm K8s được tạo bằng kubeadm. **Lưu ý:** K8s không hỗ trợ bỏ qua các phiên bản Minor trong quá trình nâng cấp. 

K8s khuyến nghị nên nâng cấp lên phiên bản Patch mới nhất càng sớm càng tốt, đồng thời đẩm bảo cụm của bạn luôn chạy một phiên bản Minor còn được hỗ trợ. 

## The first and perhaps most important step 

Trong bất kì lúc nào mà bạn muốn upgrade k8s version, điều đầu tiên và gần như bắt buộc nên làm là hãy backup toàn bộ mọi thứ (vì không ai đảm bảo việc upgrade sẽ luôn thuận lợi và dữ liệu của bạn sẽ luôn được đảm bảo)

## I. Upgrade Workflow 

Ở mức tổng quan, quy trình nâng cấp Kubernetes bằng kubeadm bao gồm các bước sau: 

1. Nâng cấp Primary Control Plane Node 
2. Nâng cấp các Control Plane Node còn lại 
3. Nâng cấp các Worker Node 

## II. Những lưu ý khi nâng cấp etcd 

Do kube-apiserver được triển khai dưới dạng Static Pod, nên nó sẽ luôn chạy, kể cả khi đã drain Control Plane Node 

Nếu quá trình kubeadm upgrade bao gồm cả việc nâng cấp etcd, thì trong lúc static Pod của etcd được khởi động lại, mọi request đang được gửi đến kube-apiserver sẽ bị tạm dừng cho đến khi etcd hoạt động trở lại 

Để giảm thiểu điều này, K8s khuyến nghị chủ động dừng kube-apiserver 1 cách an toàn vài giây trước khi chạy lệnh `kubeadm upgrade apply` 

Thực hiện trên mỗi Control Plane Node như sau: 

```bash
killall -s SIGTERM kube-apiserver   # Thực hiện graceful shutdown cho kube-apiserver

sleep 20                            # Chờ khoảng 20 giây để các request hiện tại hoàn thành

kubeadm upgrade ...                 # Thực hiện lệnh nâng cấp kubeadm
```

## III. Change The Kubernetes Package Repository 

hướng dẫn cách kích hoạt (enable) kho lưu trữ gói (package repository) tương ứng với phiên bản Minor của Kubernetes mà bạn muốn nâng cấp.

**Lưu ý:** 

- Bước này chỉ cần thực hiện khi nâng cấp từ một Minor Version sang Minor Version khác 
- Ví dụ: 
  - v1.31 -> v1.32
  - v1.32 -> v1.33
- Nếu bạn chỉ nâng cấp Patch Version trong cùng 1 Minor Version (v1.32.5 -> v1.32.7) thì không cần làm theo hướng dẫn này 
- Tuy nhiên, nếu hệ thống của bạn vẫn đang sử dụng các kho lưu trữ cũ, bạn phải chuyển sang các kho lưu trữ mới do cộng đồng quản lý trước khi tiến hành nâng cấp.
- Các kho lưu trữ cũ: `apt.kubernetes.io` hay `yum.kubernetes.io` đã hcinsh thức ngừng phát triển từ ngày 13/09/2023
- K8s khuyến nghị và yêu cầu sử dụng các kho lưu trữ mới tại `pkg.k8s.io` để cài đặt các phiên bản K8s được phát hành sau ngày 13/09/2023

### 3.0 Kiểm tra hệ thống đang sử dụng kho lưu trữ K8s nào 

Hiển thị nội dung của tệp cấu hình khai báo APT Repository của K8s: 

```bash
pager /etc/apt/sources.list.d/kubernetes.list
```

Nếu kết quả như sau: 

```bash
devops@k8s-master-01:~$ pager /etc/apt/sources.list.d/kubernetes.list
deb [signed-by=/etc/apt/keyrings/kubernetes-apt-keyring.gpg] https://pkgs.k8s.io/core:/stable:/v1.30/deb/ /
```

Thì có nghĩa là hệ thống của bạn đang sử dụng kho lưu trữ gói K8s mới

### 3.1 Chuyển sang một Kubernetes Package Repository khác 

Bước này phải được thực hiện khi nâng cấp từ một Minor Version sang Minor Version khác, nhằm giúp hệ thống có thể truy cập và cài đặt các gói phần mềm của Minor Version mới.

```bash
vim /etc/apt/sources.list.d/kubernetes.list
```

## V. Upgrading control plane nodes

Quy trình nâng cấp các Control Plane Node cần được thực hiện lần lượt từng node một. Hãy chọn 1 Control Plane Node để nâng cấp đầu tiên

### 5.0 Chạy `kubeadm upgrade`

Đối với Control Plane Node đầu tiên 

1. Nâng cấp `kubeadm`

```bash
# Thay x trong 1.31.x-* bằng phiên bản Patch mới nhất
sudo apt-mark unhold kubeadm && \
sudo apt-get update && sudo apt-get install -y kubeadm='1.31.x-*' && \
sudo apt-mark hold kubeadm
```

2. Kiểm tra phiên bản `kubeadm`

Đảm bảo rằng gói đã được tải xuống thành công và có đúng phiên bản mong muốn: 

```bash
kubeadm version
```

```bash

```

- Ở đây tôi đã tải thành công kubeadm v1.31.14 

3. Kiểm tra kế hoạch nâng cấp (Upgrade Plan) 


4. Thực hiện nâng cấp 

Chọn phiên bản Kubernetes mà bạn muốn nâng cấp, sau đó chạy lệnh tương ứng.

```bash
# Thay x bằng phiên bản Patch mà bạn đã chọn
sudo kubeadm upgrade apply v1.31.x
```

Sau khi lệnh hoàn tất, bạn sẽ thấy thông báo tương tự:

```bash
[upgrade/successful] SUCCESS! Your cluster was upgraded to "v1.31.14". Enjoy!

[upgrade/kubelet] Now that your control plane is upgraded, please proceed with upgrading your kubelets if you haven't already done so.
```

- Cụm K8s đã được nâng cấp thành công lên `v1.31.14`
- Bước tiếp theo là tiến hành nâng cấp kubelet trên các node nếu bạn chưa thực hiện 

5. Nâng cấp thủ công CNI plugin

- Hãy nâng cấp Container Network Interface (CNI) Provider Plugin theo hướng dẫn của nhà cung cấp.

- Mỗi CNI có thể có quy trình nâng cấp riêng.

- Nếu CNI của bạn được triển khai dưới dạng DaemonSet, thì không cần thực hiện bước này trên các Control Plane Node còn lại.

### 5.1 Đối với các Control Plane Node còn lại 

Quy trình gần giống với Control Plane Node đầu tiên.

Tuy nhiên, thay vì chạy:

```bash
sudo kubeadm upgrade apply
```
hãy sử dụng:

```bash
sudo kubeadm upgrade node
```
Ngoài ra:

- Không cần chạy lại: `sudo kubeadm upgrade plan`
  
- Không cần nâng cấp lại CNI Provider Plugin.

### 5.2 Drain Node 

Chuẩn bị node cho quá trình bảo trì bằng cách: 

- Đánh dấu node là Unschedulable 
- Di chuyển các workload đang chạy sang các node khác 

```bash
# Thay <node-to-drain> bằng tên node cần Drain
kubectl drain <node-to-drain> --ignore-daemonsets
```

### 5.3 Nâng cấp kubelet vs kubectl 

Sau khi Control Plane đã được nâng cấp, tiến hành nâng cấp kubelet và kubectl.

```bash
# Thay x trong 1.31.x-* bằng phiên bản Patch mới nhất
sudo apt-mark unhold kubelet kubectl && \
sudo apt-get update && sudo apt-get install -y kubelet='1.31.x-*' kubectl='1.31.x-*' && \
sudo apt-mark hold kubelet kubectl
```

### 5.4 Khởi động lại kubelet 

Sau khi nâng cấp xong, khởi động lại dịch vụ kubelet: 

```bash
sudo systemctl daemon-reload
sudo systemctl restart kubelet
```

### 5.5 Uncordon Node 

Sau khi hoàn tất việc nâng cấp và xác nhận node hoạt động bình thường, đưa node trở lại trạng thái có thể nhận Pod mới bằng cách Uncordon:

```bash
# Thay <node-to-uncordon> bằng tên node cần Uncordon
kubectl uncordon <node-to-uncordon>
```

- Node sẽ được đánh dấu là Schedulable, cho phép Kubernetes tiếp tục lập lịch (schedule) các Pod mới lên node này.

## VI. Upgrade worker nodes 

Quy trình nâng cấp các Worker Node nên được thực hiện lần lượt từng node một, bạn cần đảm bảo rằng cụm K8s vẫn còn đủ năng lực để chạy các ứng dụng hiện có, tránh việc đưa quá nhiều Worker Node vào trạng thái bảo trì cùng lúc, dẫn đến thiếu tài nguyên hoặc làm gián đoạn dịch vụ 

Bạn nên hoàn tất việc nâng cấp các Control Plane Node trước, sau đó mới tiến hành nâng cấp các Worker Node chạy Linux.

### 6.0 Thay đổi Kubernetes Package Repository 

Cũng giống như khi upgrade các Control Plane Node ở trên, ta cũng cần thay đổi Package Repo của K8s trên Node Worker về phiên bản mà mình muốn uggrade lên 

### 6.1 Nâng cấp Worker Node 

1. Nâng cấp `kubeadm`

```bash
# Thay x trong 1.31.x-* bằng phiên bản Patch mới nhất
sudo apt-mark unhold kubeadm && \
sudo apt-get update && sudo apt-get install -y kubeadm='1.31.x-*' && \
sudo apt-mark hold kubeadm
```

2. Chạy `kubeadm upgrade`

Đối với Worker Node, lệnh này sẽ nâng cấp cấu hình cục bộ (local configuration) của kubelet.

```bash
sudo kubeadm upgrade node
```

3. Drain Node 

Chuẩn bị Node trong quá trình bảo trì bằng cách: 

- Đánh dấu node là Unschedulable
- Di chuyển các workload đang chạy sang node khác 

**Lưu ý:** Lệnh này phải được thực hiện trên một Control Plane Node (Ở đây là `k8s-master-01`)

```bash
# Thực hiện lệnh này trên một Control Plane Node
# Thay <node-to-drain> bằng tên Worker Node cần Drain
kubectl drain <node-to-drain> --ignore-daemonsets
```

4. Nâng cấp `kubelet` và `kubectl`

```bash
# Thay x trong 1.31.x-* bằng phiên bản Patch mới nhất
sudo apt-mark unhold kubelet kubectl && \
sudo apt-get update && sudo apt-get install -y kubelet='1.31.x-*' kubectl='1.31.x-*' && \
sudo apt-mark hold kubelet kubectl
```

5. Khởi động lại `kubelet`

```bash
sudo systemctl daemon-reload
sudo systemctl restart kubelet
```

6. Uncordon Node 

Đưa Worker Node trở lại trạng thái hoạt động bình thường bằng cách đánh dấu node là Schedulable, cho phép Kubernetes tiếp tục lập lịch (schedule) các Pod mới lên node.

**Lưu ý:** Lệnh này phải được thực hiện trên một Control Plane Node.

```bash
# Thực hiện lệnh này trên một Control Plane Node
# Thay <node-to-uncordon> bằng tên Worker Node
kubectl uncordon <node-to-uncordon>
```

### 6.2 Nâng cấp các Worker Node còn lại với workflow như trên 

- Ở đây sau khi Drain Worker 02, hệ thống bị downtime lý do là vì khi mình upgrade k8s-worker-01 các workload đã được chuyển hết sang k8s-worker-02 rồi, do đó khi ta drain worker-02 hệ thống cần 1 khoảng thời gian để chuyển các workload sang worker-01
- Sau khoảng chừng 30s thì hệ thống đã hoạt động lại bình thường
